# 에이전틱 루프 작성 규칙

## 루프 제어 방법

- 메인세션에서 작업 진행과 루프 제어를 담당. 검증 작업은 subagent로 진행해서 컨텍스트 분리
- 에이전틱 루프 각 단계별로 스킬을 만들지 않고 스킬은 `experiment-manager` 1개만 만들기
- 사용자가 스킬 호출하면 사용자한테 현재 진행상황을 설명하고 선택지를 제시하는 작업 대시보드 역할 수행
- SKILL.md는 얇게 만들고 단계별 스크립트는 references에 작성하고 필요할때 로드하기
- state.json 이용해서 루프의 진행 상황을 기록


## 컨텍스트 로드 방법
- 처음부터 모든 컨텍스트를 로드 하지말고 작업별로 필요한 컨텍스트를 점진적으로 로드 
- 사용자가 프롬프트를 입력했을때 glossary.md 파일을 읽고 용어를 align 한다. 사용자가 사용한 용어의 의미가 명확하지 않은 경우에는 사용자한테 되묻는다.
- index.md 파일을 읽고 작업에 필요한 문서들을 선택해서 읽는다.
- **중요** 코드 그라운딩 필수. 문서는 빠르게 컨텍스트를 파악하는 목적으로 활용하고 모든 작업을 하기전에 반드시 코드를 읽고 확인한다.

## Subagent를 이용한 검증 게이트 
- 작업결과물에 대해서 subagent를 이용해서 검증
- Critical, Major, Minor 로 이슈를 분류
- 발견된 이슈는 즉시 수정
- 최대 3회까지 자동으로 수정하고 Minor 이슈만 남으면 스킵. Major 이상 이슈가 남아있으면 에스컬레이션
- 오류가 아닌 의사결정이 필요한 이슈에 대해서는 사용자에게 에스컬레이션하기

# 에이전틱 루프 워크플로우

## 0단계. bootstrap
- 프로젝트 문서, 코드 분석
- 에이전틱 루프를 사용하기 위한 문서 재구조화 및 신규 문서 작성
- 사용자가 작성한 PRD 없다면 작성하도록 가이드 제공
- 프로젝트에 에이전틱 루프 온보딩할때 최초 한번만 사용


## 0.5단계. long-term plan
- 여러 루프의 작업에 대한 장기 계획 수립
- 1단계에서 사용자 요구사항이 너무 커서 여러 루프에 걸쳐서 수행해야하는 경우 0.5 단계로 진입

## 1단계. Experiment Design

- 한 루프에서 어떤 실험을 할지 계획.
- 프로젝트 컨텍스트 파악 및 대안 제시를 위한 관련된 최신 연구논문, 깃레포, 블로그 조사
- 이번 루프에서 진행할 작업에 대한 2-3가지 선택지를 제안. 사용자 입력을 직접 받을 수 있도록 하는 선택지도 포함
- Experiment Design Doc 작성.
    - 문제정의
    - 데이터
    - 가설
    - 탐색공간 
    - 모델 평가방법
    - 성공조건 Acceptance Criteria 
    - 범위제외 Out of Scope
- 실험설계 문서를 작성하는 과정에서 의사결정이 필요한 사항들에 대해서 사용자에게 2-3개 대안을 선택하거나 응답을 요청
- subagent로 리뷰 및 자동 수정
- 사용자한테 리뷰 요청

## 2단계. Tech Design

- 실험 수행에 필요한 데이터 준비, 모델 학습 및 평가 코드를 구현
- Tech Design Spec 작성
    - Experiment Design Doc 기반으로 기술 스펙 정의
        - AS-IS / TO-BE로 정리 (**코드 그라운딩 필수**)
        - 성공조건 검증 방법 정의
            - Acceptance Criteria 를 기술적으로 재정의
            - 코드 리뷰 어떻게 할지
            - 테스트 어떻게 할지
- Implementation Plan 작성
    - Tech Design Spec 기반으로
    - 구현 부터 검증까지 구체적으로 어떤 작업을 할지. 어떤 순서로 할지 정의
    - Task spec 정의
    - Task 순서 및 의존성
- subagent 검증게이트
- 사용자한테 리뷰 요청 


## 3단계. 구현

- Tech Design Spec, Implementation Plan 대로 순차적으로 TDD 방식으로 구현
- subagent 검증게이트

## 4단계. 실험

- 실험 수행
    - 데이터 준비
    - 학습
    - 평가
- 실험 결과 보고서 작성
    - markdown으로 먼저 작성
    - html로 사용자가 보기 편하게 작성
- 사용자한테 리뷰 받기

## 5단계. 마무리

- gabage collection
    - 이번 루프 작업으로 인해서 필요없어진 코드 및 문서 정리
- LLM wiki 업데이트
    - subagent 이용해서 업데이트하고 검증하기. 문서 디렉토리 단위로 병렬로 수행
    - 코드와 human-source 문서들 수정사항 기반으로 클레임 단위로 효율적으로 검증하고 업데이트
- tools & hooks 업데이트
    - 이번 loop의 작업내용을 리뷰하고 필요한 tools와 hooks를 업데이트
    - 자주 사용하는 꼭 필요한 tools와 hooks만 만들기. tools 와 hooks 과잉 방지


# LLM wiki

목적: 프로젝트 컨텍스트 정보를 지속적으로 업데이트하고 에이전트가 쉽게 검색하고 활용할 수 있도록 한다.

### 문서 구조화

문서 디렉토리를 관리주체, 사용기한, 주제별로 적절히 분리해서 구조화하기

관리 주체에 따른 분류
1. human-authored: 사람이 직접 작성한 문서
2. agent-generated: 에이전트가 작성한 문서

사용기한에 따른 분류
1. 루프 안에서 생성되고 소비되는 문서
2. 프로젝트에서 지속적으로 관리되는 문서

네비게이션 문서
- glossary.md: 사람과 코딩 에이전트의 소통을 위한 용어집 
- index.md: 에이전트가 필요한 문서를 찾는 네비게이션 역할

### LLM wiki 규칙

- agent가 작성하는 문서들은 claim 단위로 파일 출처를 명시하기
- 자주 사용하는 문서들은 skill 디렉토리 templates에 템플릿화 해서 관리
    - Project Requirements Doc
    - Experiment Design Doc
    - Tech Design Spec 
    - Task Spec
    - Implementation Plan
    - Experiment Report


## 주의사항
- .claude의 모든 파일과 CLAUDE.md는 영어로 작성하기
- glossary.md 제외한 모든 문서와 코드는 영어로 작성
- 사용자와 의사소통할때는 기본적으로 한국어 사용하기. 전문용어는 영어 사용
