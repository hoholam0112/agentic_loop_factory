# 에이전틱 루프 작성 프롬프트 

## 목적과 원칙
- 이 문서는 실험 자동화를 위한 에이전틱 루프의 운영 원칙과 워크플로우를 정의한다.
- 각 단계는 목적 · 산출물 · 완료조건을 가진다. 완료조건을 만족하는 한 구현 방식은 에이전트가 결정한다.
- **코드 그라운딩 필수 (전 단계 공통 원칙).** 문서는 빠르게 컨텍스트를 파악하는 목적으로만 활용하고, 판단·구현·검증 전에는 반드시 실제 코드를 읽고 확인한다. 문서와 코드가 다르면 코드를 신뢰한다.

## 실행 메커니즘

### 루프 제어
- 메인세션이 작업 진행과 루프 제어를 담당한다. 검증 등 컨텍스트 오염 우려가 있는 작업은 subagent로 분리한다.
- 스킬은 `experiment-loop` **1개만** 만든다. 단계별로 스킬을 나누지 않는다.
- SKILL.md는 얇게 유지하고, 단계별 상세 절차는 references에 두고 필요할 때 로드한다. 각 단계 진입 시 해당 단계의 절차를 로드한다.
- `state.json`으로 루프 진행 상황을 기록한다. **중단 후 재개와 다음 루프 인계에 필요한 최소 정보**를 남기는 것을 목표로 하며, 구체적 필드는 에이전트가 정한다.
- 사용자가 스킬을 호출하면 **작업 대시보드** 역할을 한다: 먼저 `state.json`을 확인해 현재 위치를 파악하고, 진행 상황을 설명한 뒤 선택지를 제시한다. 온보딩 전(state 없음)이면 0단계 bootstrap으로 안내하고, 진행 중이면 해당 지점부터 이어간다.

### 컨텍스트 로드
- 처음부터 모든 컨텍스트를 로드하지 않고, 작업별로 필요한 컨텍스트를 점진적으로 로드한다.
- 사용자 입력을 받으면 `glossary.md`를 읽고 용어를 align한다. 사용자가 쓴 용어의 의미가 명확하지 않으면 되묻는다.
- `index.md`를 읽고 작업에 필요한 문서를 선택해 읽는다.

### 검증 게이트 (subagent)
- 작업 결과물은 subagent를 이용해 검증한다.
- 이슈는 Critical / Major / Minor로 분류한다. 분류 기준은 **실험 결과의 유효성과 재현성에 미치는 영향의 크기**로 판단한다.
- 발견된 이슈는 즉시 수정한다. 최대 3회까지 자동으로 수정하고, Minor만 남으면 스킵한다. Major 이상이 남으면 에스컬레이션한다.
- 오류가 아니라 **의사결정이 필요한 이슈**는 사용자에게 에스컬레이션한다.
- 에스컬레이션은 루프를 멈추고, 상황과 2~3개 선택지를 사용자에게 제시하는 것으로 한다.

## 에이전틱 루프 워크플로우

각 단계는 완료조건을 만족하면 다음 단계로 넘어가며, 사용자 리뷰가 명시된 단계는 사용자 승인을 완료조건으로 한다.
각 단계별로 subagent 검증 게이트를 완료하면 사용자에게 각 단계에서 진행한 작업에 대해서 요약해서 설명한다.
사용자에게 설명할 때(작업 전후, 게이트 요약 포함)는 쉽게 읽히도록 한다. 구체적으로:
- 결론을 먼저 말하고 이유는 짧게 붙인다.
- 한 문장에는 한 가지 생각만 담고, 문장을 짧게 쓴다.
- 전문용어는 꼭 필요할 때만 쓰고, 처음 쓸 때 짧게 뜻을 붙인다.
- 우리말이 있는데 영어를 소리 나는 대로 옮긴 말(예: 추적성, 레버리지)은 쓰지 않고 쉬운 우리말로 바꾼다.
- 추상적인 명사보다 구체적인 동사를 쓴다.

### 0단계. bootstrap
- 프로젝트 문서·코드 분석
- 에이전틱 루프를 사용하기 위한 문서 재구조화 및 신규 문서 작성
- 사용자가 작성한 PRD가 없다면 작성하도록 가이드 제공
- 프로젝트에 에이전틱 루프를 온보딩할 때 최초 1회만 사용

### 0.5단계. long-term plan
- 여러 루프에 걸친 작업의 장기 계획 수립
- 1단계에서 사용자 요구사항이 한 루프로 담기 어렵다고 판단되면 진입한다. 판단이 애매하면 사용자에게 확인한다.

### 1단계. Experiment Design
- 한 루프에서 어떤 실험을 할지 계획한다.
- 프로젝트 컨텍스트 파악 및 대안 제시를 위해 관련 최신 연구논문과 오픈소스 레포를 조사한다.
- 지난 실험 에러 되돌아보기: 이전 루프의 실험보고서(에러분석)와 관련 지식·결정 기록을 읽고, 주요 오류 패턴을 정리해 이번 루프에서 무엇을 다룰지 정한다. 이는 가설과 탐색공간의 근거가 된다. 첫 루프라 이전 보고서가 없으면 그렇게 명시한다.
- 이번 루프에서 진행할 작업에 대해 2~3가지 선택지를 제안한다. 사용자가 직접 입력할 수 있는 선택지도 포함한다.
- Experiment Design Doc 작성하기.
- Experiment Design Doc 각 섹션에 대해서 사용자에게 2~3개 선택지를 가진 질문을 제시한다.
- subagent로 리뷰 및 자동 수정 → 사용자에게 리뷰 요청

### 2단계. Tech Design
- 실험 수행에 필요한 데이터 준비, 모델 학습·평가 코드를 구현하기 위한 설계.
- Tech Design Spec 작성 (**코드 그라운딩 필수**)
  - Experiment Design Doc 기반으로 기술 스펙 정의: AS-IS / TO-BE
  - 성공조건 검증 방법 정의: Acceptance Criteria의 기술적 재정의, 코드 리뷰 방법, 테스트 방법
- Implementation Plan 작성
  - Tech Design Spec 기반으로 구현부터 검증까지 어떤 작업을 어떤 순서로 할지 정의
  - Task spec 정의, Task 순서 및 의존성
- subagent 검증 게이트 → 사용자에게 리뷰 요청

### 3단계. 구현
- Tech Design Spec, Implementation Plan에 따라 순차적으로 TDD 방식으로 구현
- subagent 검증 게이트

### 4단계. 실험
- 실험 수행: 데이터 준비 / 학습 / 평가
- 실험 결과 보고서 작성: markdown으로 먼저 작성 후, 사용자가 보기 편하도록 html로 작성
- subagent 검증 게이트: 모든 수치가 출력 파일로 추적되는지, acceptance criterion마다 pass/fail이 평가됐는지, 한계가 명시됐는지에 더해 **독립성**(코드·문서를 열지 않고 보고서만으로 이해되는지)과 **충분한 깊이**(자리표시자 없이 각 섹션이 실제 내용으로 채워졌는지)를 검사한다.
- 사용자에게 리뷰 받기

### 5단계. 마무리
- garbage collection: 이번 루프로 불필요해진 코드·문서 정리
- LLM wiki 업데이트: subagent로 업데이트·검증하며 문서 디렉토리 단위로 병렬 수행. 코드와 human-source 문서의 수정사항 기반으로 claim 단위로 효율적으로 검증·업데이트
- loops에 있는 내용중 persistant로 승격 필요한 것들 승격시키기
- tools & hooks 업데이트: 이번 루프 작업을 리뷰해 필요한 tools·hooks를 업데이트. 자주 쓰는 꼭 필요한 것만 만들고 과잉을 방지한다.

## LLM wiki
LLM wiki에 대한 프롬프트를 `references`에 별도로 작성한다.

### 작성원칙
- 프로젝트 컨텍스트 정보를 지속적으로 업데이트하고, 에이전트가 쉽게 검색·활용할 수 있도록 한다.
- 에이전트가 작성하는 문서는 claim 단위로 source of truth를 파일 단위로 명시한다.
- 각 단계가 끝났을때 llm-wiki를 업데이트한다.
- 자주 쓰는 문서는 skill 디렉토리 templates에 템플릿화해 관리한다: Project Requirements Doc / Experiment Design Doc / Tech Design Spec / Task Spec / Implementation Plan / Experiment Report

### 문서 구조화
문서 디렉토리를 관리주체 · 사용기한 · 주제별로 적절히 분리해 구조화한다.

  llm-wiki
├── index.md              # 에이전트 진입점. 전체 지도 (자동 갱신)
├── glossary.md           # 사람과 에이전트의 소통을 위한 용어집
├── CONVENTIONS.md        # 이 위키의 규칙 (권한·우선순위·frontmatter 스펙)
│
├── human/                # ✍️ 사람이 소유.
│   ├── raw/              #   사람이 직접 작성한 원본 (최고 권위). 에이전트는 편집 불가
│
├── agent/                # 🤖 에이전트가 읽고 씀. 사람은 안 봄
│   ├── knowledge/        #   지속 관리되는 학습·정리 (persistent)
│   ├── decisions/        #   결정 기록 (ADR). 왜 그렇게 했는가
│   └── loops/            #   루프 안에서 생성·소비 (ephemeral, GC 대상)
│
└── shared/               # 👀 에이전트가 쓰고 사람이 봄
    └── ...               #   리포트·요약 등 human-view 산출물


## 주의사항
- `.claude`의 모든 파일과 `CLAUDE.md`는 영어로 작성한다.
- `glossary.md`를 제외한 모든 문서와 코드는 영어로 작성한다.
- 사용자와 소통할 때는 기본적으로 한국어를 사용하고, 전문용어는 영어를 사용한다.
- 어려운 단어의 사용은 지양하고 최대한 쉬운 표현을 사용한다. 전문용어는 반드시 필요할때만 사용한다.


# 문서템플릿

## Experiment Design Doc

### 작성원칙
- 자세하지만 쉽게 작성하기. 어려운 단어는 사용하지 않는다. 전문용어는 꼭 필요한 수준으로만 사용한다. 학부생 수준에서 무리없이 이해할 수 있는 수준으로 작성한다.

### 구성요소
- 문제정의: 어떤 문제를 해결하려고 하는건지
- 데이터: 어떤 데이터를 어떻게 수집하고 어떻게 전처리하고 어떻게 구성할 건지
- 외부조사: 관련된 최신 논문과 오픈소스 레포지토리 조사. 조사 결과는 파일로 별도로 작성하고 요약과 참조만
- 지난 에러 되돌아보기: 이전 루프의 실험보고서(에러분석)와 지식·결정 기록에서 주요 오류 패턴을 정리하고, 이번 루프에서 다룰 것과 뺄 것을 명시. 출처 표기. 첫 루프면 이전 보고서 없음을 명시
- 가설: 이 실험을 통해서 어떤 가설을 입증하려고하는 건지
- 탐색공간: 탐색해볼 수 있는 구성요소들에 대해서 정의하고 대안 비교
- 평가방법: 평가지표 
- 제약사항: 지연속도, 요청량, 비용, 법적이슈 등
- 성공조건: 실험의 성공을 판단할 수 있는 조건들. 정확도, 구현완성도 등 
- 범외제외: 이번 실험 범위에서 제외할 것들 명시. 한 루프에서 과도하게 많은 작업하는 것 방지


## 실험보고서
실험보고서를 html로 작성하는 프롬프트를 `references`에 별도로 작성한다.

### 작성원칙
- 실험보고서는 자세하지만 쉽게 만든다. 꼭 필요한 전문용어 아니면 어려운 단어의 사용은 지양한다. 학부생 수준에서 무리 없이 읽고 이해할 수 있도록 작성한다.
- 독립적으로 작성한다. 다른 코드나 문서를 참고하지 않고 이 보고서만 읽어도 실험 전체를 이해할 수 있어야 한다. "train.py 참고", "기술설계 참고"처럼 이해를 외부 파일로 미루지 않고, 필요한 맥락(데이터·방법·지표 의미)을 보고서 안에 다시 설명한다. 인용한 파일은 이미 서술한 주장의 근거일 뿐, 읽어야만 이해되는 포인터가 아니다.
- 자리표시자 없이 각 섹션을 실제 내용으로 채운다: 데이터는 실제 샘플과 split 크기, 모델은 pseudo code·수식(기호 정의 포함)·프롬프트 원문, 실험히스토리는 채워진 표와 변경 내역 서술, 에러분석은 20개 이상의 실제 케이스를 담는다.
- 시니어 프론트엔드로써 HTML, CSS, JavaScript를 사용해서 깔끔하고 세련된 디자인의 보고서를 작성한다.
- 실험 결과를 직관적으로 쉽게 이해할 수 있도록 그래프를 적절히 활용한다.
- 최종적인 결과는 하나의 독립적인 HTML 파일로 만든다. 별도 설치 없이 브라우저에서 바로 열려야한다.
- 개요, 데이터, 모델, 실험히스토리, 에러분석을 각각 별도 탭으로 만든다.

### 실험보고서 구성요소
  - 개요: 배경, 문제정의, 데이터, 실험히스토리, 정확도, 성능지표
  - 데이터: 데이터 수집 / 전처리 / 구성 / 샘플데이터
  - 모델: 실험한 모델에 대한 자세한 설명. pseudo code와 수식으로 자세하게 설명. 프롬프트는 원문 포함
  - 실험히스토리: 지금까지 어떤 실험들을 진행해왔고 결과가 어땠는지 표와 함께 설명
  - 에러분석: 오류케이스와 정답케이스를 최소 20-30개 샘플링해서 데이터, 정답, 모델결과를 한눈에 들어오도록 구성
