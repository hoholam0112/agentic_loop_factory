# 에이전틱 루프 작성 규칙

## 목적과 원칙
- 이 문서는 실험 자동화를 위한 에이전틱 루프의 운영 원칙과 워크플로우를 정의한다.
- 에이전트에게 **"무엇을, 왜"** 를 전달하고 **"어떻게"** 는 에이전트의 판단에 최대한 위임한다. 세부 절차를 지시하기보다 의도와 완료조건을 명확히 하는 것을 우선한다.
- 각 단계는 목적 · 산출물 · 완료조건을 가진다. 완료조건을 만족하는 한 구현 방식은 에이전트가 결정한다.
- **코드 그라운딩 필수 (전 단계 공통 원칙).** 문서는 빠르게 컨텍스트를 파악하는 목적으로만 활용하고, 판단·구현·검증 전에는 반드시 실제 코드를 읽고 확인한다. 문서와 코드가 다르면 코드를 신뢰한다.

## 실행 메커니즘

### 루프 제어
- 메인세션이 작업 진행과 루프 제어를 담당한다. 검증 등 컨텍스트 오염 우려가 있는 작업은 subagent로 분리한다.
- 스킬은 `experiment-loop` **1개만** 만든다. 단계별로 스킬을 나누지 않는다.
- SKILL.md는 얇게 유지하고, 단계별 상세 절차는 references에 두고 필요할 때 로드한다. 각 단계 진입 시 해당 단계의 절차를 로드한다.
- `state.json`으로 루프 진행 상황을 기록한다. **중단 후 재개와 다음 루프 인계에 필요한 최소 정보**를 남기는 것을 목표로 하며, 구체적 필드는 에이전트가 정한다.
- 사용자가 스킬을 호출하면 **작업 대시보드** 역할을 한다: 먼저 `state.json`을 확인해 현재 위치를 파악하고, 진행 상황을 설명한 뒤 선택지를 제시한다. 온보딩 전(state 없음)이면 0단계 bootstrap으로 안내하고, 진행 중이면 해당 지점부터 이어간다.

### 컨텍스트 로드
- 처음부터 모든 컨텍스트를 로드하지 않고, 작업별로 필요한 컨텍스트를 점진적으로 로드한다.
- 사용자 입력을 받으면 `glossary.md`를 읽고 용어를 align한다. 사용자가 쓴 용어의 의미가 명확하지 않으면 되묻는다.
- `index.md`를 읽고 작업에 필요한 문서를 선택해 읽는다.

### 검증 게이트 (subagent)
- 작업 결과물은 subagent를 이용해 검증한다.
- 이슈는 Critical / Major / Minor로 분류한다. 분류 기준은 **실험 결과의 유효성과 재현성에 미치는 영향의 크기**로 판단한다.
- 발견된 이슈는 즉시 수정한다. 최대 3회까지 자동으로 수정하고, Minor만 남으면 스킵한다. Major 이상이 남으면 에스컬레이션한다.
- 오류가 아니라 **의사결정이 필요한 이슈**는 사용자에게 에스컬레이션한다.
- 에스컬레이션은 루프를 멈추고, 상황과 2~3개 선택지를 사용자에게 제시하는 것으로 한다.

## 에이전틱 루프 워크플로우

각 단계는 완료조건을 만족하면 다음 단계로 넘어가며, 사용자 리뷰가 명시된 단계는 사용자 승인을 완료조건으로 한다.

### 0단계. bootstrap
- 프로젝트 문서·코드 분석
- 에이전틱 루프를 사용하기 위한 문서 재구조화 및 신규 문서 작성
- 사용자가 작성한 PRD가 없다면 작성하도록 가이드 제공
- 프로젝트에 에이전틱 루프를 온보딩할 때 최초 1회만 사용

### 0.5단계. long-term plan
- 여러 루프에 걸친 작업의 장기 계획 수립
- 1단계에서 사용자 요구사항이 한 루프로 담기 어렵다고 판단되면 진입한다. 판단이 애매하면 사용자에게 확인한다.

### 1단계. Experiment Design
- 한 루프에서 어떤 실험을 할지 계획한다.
- 프로젝트 컨텍스트 파악 및 대안 제시를 위해 관련 최신 연구논문·깃레포·블로그를 조사한다.
- 이번 루프에서 진행할 작업에 대해 2~3가지 선택지를 제안한다. 사용자가 직접 입력할 수 있는 선택지도 포함한다.
- Experiment Design Doc 작성: 문제정의 / 데이터 / 가설 / 탐색공간 / 모델 평가방법 / 성공조건(Acceptance Criteria) / 범위제외(Out of Scope)
- 작성 과정에서 의사결정이 필요한 사항은 사용자에게 2~3개 대안을 제시하거나 응답을 요청한다.
- subagent로 리뷰 및 자동 수정 → 사용자에게 리뷰 요청

### 2단계. Tech Design
- 실험 수행에 필요한 데이터 준비, 모델 학습·평가 코드를 구현하기 위한 설계.
- Tech Design Spec 작성 (**코드 그라운딩 필수**)
  - Experiment Design Doc 기반으로 기술 스펙 정의: AS-IS / TO-BE
  - 성공조건 검증 방법 정의: Acceptance Criteria의 기술적 재정의, 코드 리뷰 방법, 테스트 방법
- Implementation Plan 작성
  - Tech Design Spec 기반으로 구현부터 검증까지 어떤 작업을 어떤 순서로 할지 정의
  - Task spec 정의, Task 순서 및 의존성
- subagent 검증 게이트 → 사용자에게 리뷰 요청

### 3단계. 구현
- Tech Design Spec, Implementation Plan에 따라 순차적으로 TDD 방식으로 구현
- subagent 검증 게이트

### 4단계. 실험
- 실험 수행: 데이터 준비 / 학습 / 평가
- 실험 결과 보고서 작성: markdown으로 먼저 작성 후, 사용자가 보기 편하도록 html로 작성
- 사용자에게 리뷰 받기

### 5단계. 마무리
- garbage collection: 이번 루프로 불필요해진 코드·문서 정리
- LLM wiki 업데이트: subagent로 업데이트·검증하며 문서 디렉토리 단위로 병렬 수행. 코드와 human-source 문서의 수정사항 기반으로 claim 단위로 효율적으로 검증·업데이트
- tools & hooks 업데이트: 이번 루프 작업을 리뷰해 필요한 tools·hooks를 업데이트. 자주 쓰는 꼭 필요한 것만 만들고 과잉을 방지한다.

## LLM wiki
목적: 프로젝트 컨텍스트 정보를 지속적으로 업데이트하고, 에이전트가 쉽게 검색·활용할 수 있도록 한다.

### 문서 구조화
문서 디렉토리를 관리주체 · 사용기한 · 주제별로 적절히 분리해 구조화한다.
- 관리 주체: (1) human-authored — 사람이 직접 작성 (2) agent-generated — 에이전트가 작성
- 사용 기한: (1) 루프 안에서 생성·소비되는 문서 (2) 프로젝트에서 지속 관리되는 문서
- 네비게이션 문서
  - `glossary.md`: 사람과 코딩 에이전트의 소통을 위한 용어집
  - `index.md`: 에이전트가 필요한 문서를 찾는 네비게이션

### 규칙
- 에이전트가 작성하는 문서는 claim 단위로 파일 출처를 명시한다.
- 자주 쓰는 문서는 skill 디렉토리 templates에 템플릿화해 관리한다: Project Requirements Doc / Experiment Design Doc / Tech Design Spec / Task Spec / Implementation Plan / Experiment Report

## 주의사항
- `.claude`의 모든 파일과 `CLAUDE.md`는 영어로 작성한다.
- `glossary.md`를 제외한 모든 문서와 코드는 영어로 작성한다.
- 사용자와 소통할 때는 기본적으로 한국어를 사용하고, 전문용어는 영어를 사용한다.
